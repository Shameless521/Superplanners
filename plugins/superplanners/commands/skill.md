# SuperPlanners - Skill 助手

为当前项目执行一站式 Skill 管理：扫描项目 → 全面诊断 → 生成行动计划 → 用户确认 → 执行变更 → 复审验证。

如果 `{{{skill_name}}}` 不为空，则只针对该名称的 Skill 进行诊断和处理。

## 第一步：项目扫描

### 1.1 扫描已有 Skill

扫描以下目录，收集所有已有的 Skill：

```
1. .claude/skills/          ← 项目级 Skill（优先级最高）
2. .claude/commands/        ← 项目级 Command
```

对每个发现的 Skill，收集：
- `SKILL.md` 的 frontmatter（name, description 及其他字段）
- `SKILL.md` body 的行数和结构
- 子目录情况（scripts/, references/, assets/）
- 引用文件列表及大小

### 1.2 扫描项目上下文

分析项目整体情况，为覆盖率分析和 Skill 生成提供依据：

1. **技术栈识别**：扫描 `package.json`、`requirements.txt`、`go.mod`、`Cargo.toml`、`Gemfile`、`pubspec.yaml` 等依赖文件，提取核心框架和库
2. **项目结构**：识别 src/、lib/、tests/、scripts/、api/ 等关键目录及其子模块
3. **现有工作流**：检查 `.github/workflows/`、`Makefile`、`scripts/` 中的自动化流程
4. **CLAUDE.md 分析**：读取项目 CLAUDE.md 了解编码规范、约定和偏好
5. **工作模式识别**：从 git log 最近 20 条提交识别重复的工作模式（如频繁新增页面、频繁修复某类 bug）

## 第二步：全面诊断

对扫描结果进行三维诊断，全部为只读操作。

### Part 1：质量审计（已有 Skill 的规范合规性）

对每个已有 Skill 逐项检查：

#### Frontmatter 检查

| # | 检查项 | 合规标准 | 严重度 |
|---|--------|---------|--------|
| F1 | name 格式 | 仅含小写字母、数字、连字符；不以 `-` 开头或结尾；不含 `--`；长度 ≤64 字符 | 错误 |
| F2 | name 保留词 | 不含 `anthropic`、`claude` | 错误 |
| F3 | description 非空 | description 字段存在且非空字符串 | 错误 |
| F4 | description 长度 | ≤1024 字符 | 错误 |
| F5 | description 无 XML 标签 | 不含 `<` 或 `>` 字符 | 错误 |
| F6 | description 第三人称 | 不以 "I can"、"You can"、"我" 开头 | 警告 |
| F7 | description 含触发信息 | 同时包含"做什么"和"何时使用"两类信息 | 警告 |
| F8 | description 含具体关键词 | 包含与功能相关的具体名词/动词，而非泛泛描述 | 建议 |

#### Body 检查

| # | 检查项 | 合规标准 | 严重度 |
|---|--------|---------|--------|
| B1 | 行数上限 | SKILL.md body ≤500 行 | 警告 |
| B2 | 祈使语气 | 指令使用祈使句（动词开头），不出现 "You should"、"You can" | 建议 |
| B3 | 无 "When to Use" | body 中不含 "When to Use" 或 "何时使用" 段落（这属于 description） | 警告 |
| B4 | 引用深度 | 引用文件最多一层（SKILL.md → file.md），不出现嵌套引用 | 警告 |
| B5 | 有实质内容 | body 不为空，且包含可操作指令 | 错误 |

#### 目录结构检查

| # | 检查项 | 合规标准 | 严重度 |
|---|--------|---------|--------|
| D1 | SKILL.md 存在 | skill 目录下必须有 SKILL.md 文件 | 错误 |
| D2 | 描述性文件名 | 引用文件使用有意义的名称（不是 doc1.md、file2.md） | 建议 |
| D3 | 大文件有目录 | 超过 100 行的引用文件顶部有目录（TOC） | 建议 |
| D4 | 引用文件可达 | SKILL.md 中引用的文件全部实际存在 | 错误 |

#### 渐进加载评估

| # | 检查项 | 合规标准 | 严重度 |
|---|--------|---------|--------|
| P1 | Body 合理 | SKILL.md body < 5000 tokens（约 300 行以内为佳） | 建议 |
| P2 | 详细内容拆分 | 超过 500 行的内容使用 references/ 拆分 | 警告 |
| P3 | 按需引用 | SKILL.md 中对引用文件有明确的"何时读取"指引 | 建议 |

### Part 2：覆盖率分析（项目还缺什么 Skill）

将项目扫描结果与已有 Skill 进行对比，从三个维度分析缺口：

#### 技术栈覆盖

对照依赖文件中的核心框架/库，检查是否有对应的开发规范或工作流 Skill：

| 检查逻辑 | 举例 |
|---------|------|
| 核心框架有无对应 Skill | 项目用了 Next.js 但无页面开发 Skill |
| 数据层有无对应 Skill | 项目用了 Prisma 但无数据库迁移 Skill |
| 测试框架有无对应 Skill | 项目有 vitest 但无测试规范 Skill |

#### 工作模式覆盖

对照 git log 识别的高频操作模式，检查是否有对应 Skill：

| 检查逻辑 | 举例 |
|---------|------|
| 高频 feat 提交类型 | 频繁 "feat: 新增 xxx 页面" → 缺页面开发 Skill |
| 高频 fix 提交类型 | 频繁修复同类 bug → 缺调试/规范 Skill |
| 重复性操作 | 频繁手动执行部署 → 缺部署流程 Skill |

#### 模块覆盖

对照 src/ 下的核心模块目录，检查关键模块是否有对应 Skill：

| 检查逻辑 | 举例 |
|---------|------|
| 核心业务模块 | 有 `src/payment/` 但无支付相关 Skill |
| 公共模块 | 有 `src/utils/` 但无工具函数规范 Skill |
| 新增模块 | 近期提交新增了模块目录，尚无覆盖 |

### Part 3：时效性检查（哪些 Skill 可能过时了）

对每个已有 Skill，检查其内容与项目当前状态是否匹配：

| # | 检查项 | 合规标准 | 严重度 |
|---|--------|---------|--------|
| T1 | 引用路径可达 | Skill 中提到的项目文件路径仍然存在 | 错误 |
| T2 | 版本匹配 | Skill 中提到的框架/库版本与当前依赖文件一致 | 警告 |
| T3 | 模块引用有效 | Skill 中提到的模块/类/函数仍存在于项目中 | 警告 |
| T4 | 工作流匹配 | Skill 中描述的 CI/CD 或脚本流程与当前一致 | 建议 |
| T5 | description 覆盖当前场景 | description 的触发词是否覆盖了项目当前的技术栈和模块 | 建议 |

## 第四步：生成行动计划

根据诊断结果，生成**统一的行动计划**，包含所有需要的操作。

### 4.1 需要优化的 Skill（质量问题 + 时效性问题）

对每个有问题的 Skill，给出修改前后对比：

```markdown
#### 优化项 #1：auth-flow — description 缺少触发信息 (F7) + 未覆盖 OAuth (T5)

**当前**：
description: "用户登录流程管理"

**建议修改为**：
description: "用户认证流程管理工具。处理登录、注册、OAuth 授权等认证场景。使用场景：(1) 实现登录/注册功能, (2) 接入 OAuth 第三方登录, (3) 处理 token 刷新和会话管理。触发词：登录、注册、认证、OAuth、token、session"

#### 优化项 #2：auth-flow — 引用路径失效 (T1)

**当前**：参见 `src/auth/login.ts`
**建议修改为**：参见 `src/auth/sign-in.ts`
```

### 4.2 需要补充的 Skill（覆盖率缺口）

对每个建议补充的 Skill，给出完整的生成方案：

```markdown
#### 新增 Skill #1：prisma-migration

**理由**：项目使用 Prisma，git log 中有 5/20 次提交涉及数据库迁移，但无对应 Skill。
**优先级**：高（高频操作 + 易出错）

**description 草案**：
"Prisma 数据库迁移工作流。安全执行 schema 变更和数据迁移。使用场景：(1) 修改数据库 schema, (2) 执行数据迁移, (3) 处理迁移冲突。触发词：migration、迁移、schema、数据库变更、Prisma"

**body 结构**：工作流式（步骤：schema 修改 → 生成迁移 → 审查 SQL → 执行 → 验证）
**预计目录**：
  .claude/skills/prisma-migration/
  ├── SKILL.md
  └── references/migration-patterns.md
```

### 4.3 行动汇总

```markdown
### 汇总

| 操作类型 | 数量 | 详情 |
|---------|------|------|
| 优化已有 Skill | 2 | auth-flow（description + 路径修复） |
| 新增 Skill | 2 | prisma-migration, payment-flow |
| 预计变更文件数 | 5 | 修改 1 个 SKILL.md + 新建 2 个 Skill 目录 |
```

## 第五步：用户确认后执行

**展示完整的行动计划后暂停，等用户确认。**

用户可以：
- **全部接受**：批量执行所有操作
- **逐项选择**：选择执行部分操作
- **全部跳过**：仅保留诊断报告，不做任何修改

用户确认后，按以下顺序执行：

### 5.1 优化已有 Skill

对每个需要优化的 Skill：
1. 按行动计划中的对比方案修改文件
2. 修改后立即对该 Skill 重新执行质量审计，确认问题已修复

### 5.2 生成新 Skill

对每个需要新增的 Skill：

1. 创建目录 `.claude/skills/{skill_name}/`
2. 按以下模板生成 `SKILL.md`：

```yaml
---
name: {skill_name}
description: "{一句话功能说明}。{核心价值}。使用场景：(1){场景1}, (2){场景2}, (3){场景3}。触发词：{词1}、{词2}、{词3}"
---
```

```markdown
# {Skill 标题}

{一段话概述核心功能和适用场景}

## 核心流程

### 步骤 1：{动词开头的标题}

{具体指令和操作说明}

### 步骤 2：{动词开头的标题}

{具体指令和操作说明}

## 参考资料

- [详细文档](references/xxx.md)（如有）

## 输出格式

{明确的输出模板或示例}
```

3. 按需创建 scripts/、references/、assets/ 子目录和文件
4. 生成后对该 Skill 执行完整质量审计，确保零错误、零警告

### body 结构模式选择

根据 Skill 类型选择最合适的结构：

| 模式 | 适用场景 | 特征 |
|------|---------|------|
| **工作流式** | 多步骤顺序操作（部署、迁移） | 编号步骤 + 检查点 |
| **任务式** | 单一目标任务（代码审查） | 输入 → 处理 → 输出 |
| **参考式** | 知识/规范/约定 | 分类索引 + 按需引用 |
| **能力式** | 扩展通用能力（解释代码） | 行为准则 + 示例 |

## 第六步：执行后复审

所有变更完成后，对**所有受影响的 Skill**（包括优化的和新生成的）执行一轮完整的质量审计。

输出复审结果：

```markdown
## 复审结果

| Skill | 操作 | 审计结果 |
|-------|------|---------|
| auth-flow | 优化 | ✅ 零错误、零警告 |
| prisma-migration | 新增 | ✅ 零错误、零警告 |
| payment-flow | 新增 | ✅ 零错误、零警告 |

全部通过，本次 Skill 维护完成。
```

如果复审发现问题，立即修复后再次复审，直到全部通过。

## 附录：Skill 官方规范速查

### 命名规则（name 字段）

- 仅允许：小写字母 `a-z`、数字 `0-9`、连字符 `-`
- 长度上限：64 字符
- 禁止：以 `-` 开头或结尾、连续 `--`、包含 `anthropic` 或 `claude`

### 描述规则（description 字段）

- 长度上限：1024 字符
- 禁止包含 `<` 或 `>` 字符
- 使用第三人称（不用 "I"、"You"、"我"）
- 必须同时说明"做什么"和"何时使用"
- **description 是触发机制的唯一决定因素**，body 中的 "When to Use" 无效

### 推荐 description 格式

```
"[一句话功能说明]。[核心价值]。使用场景：(1)..., (2)..., (3)...。触发词：xxx、yyy、zzz"
```

### Body 规则

- 使用祈使语气（动词开头的指令句）
- 总行数控制在 500 行以内
- 详细内容拆到 references/ 目录
- 不要在 body 中写 "When to Use"（属于 description 的职责）
- 引用文件保持一层深度，不要嵌套引用

### 渐进加载层级

| 层级 | 内容 | 加载时机 | 预算 |
|------|------|---------|------|
| L1 | name + description | 始终加载 | ~100 tokens |
| L2 | SKILL.md body | 触发时加载 | <5000 tokens |
| L3 | 引用文件/脚本 | 按需加载 | 无限制 |

### 目录结构

```
skill-name/
├── SKILL.md          # 必填：主指令文件
├── scripts/          # 可选：可执行脚本（执行而非读取）
├── references/       # 可选：参考文档（读取到上下文）
└── assets/           # 可选：模板、数据等输出资源
```
