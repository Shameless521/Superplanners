# SuperPlanners - 代码优化分析

扫描项目代码，从多个维度发现优化点，将确认的优化项转化为 SuperPlanners 任务计划进行管理。

如果 `{{{scope}}}` 不为空，则只扫描指定目录或文件范围。
如果未指定 `dimensions`，默认使用 5 个核心维度。

## 第一步：环境探测

### 1.1 识别项目技术栈

扫描以下文件确定技术栈：

- `package.json` / `package-lock.json` / `yarn.lock` / `pnpm-lock.yaml`（Node.js）
- `requirements.txt` / `pyproject.toml` / `setup.py`（Python）
- `go.mod`（Go）
- `Cargo.toml`（Rust）
- `pubspec.yaml`（Flutter/Dart）
- `Gemfile`（Ruby）
- `pom.xml` / `build.gradle`（Java）

记录：语言、框架、构建工具、测试框架、核心依赖。

### 1.2 确定扫描范围

- 有 `scope` 参数：只扫描指定目录/文件
- 无 `scope` 参数：扫描 `src/`、`lib/`、`app/` 等源码目录（排除 `node_modules/`、`dist/`、`.git/` 等）

### 1.3 确定扫描维度

解析 `dimensions` 参数：

| 值 | 含义 |
|---|---|
| 不传 或 `core` | 5 个核心维度 |
| `all` | 全部 10 个维度 |
| `dim1,dim2,...` | 指定维度（如 `duplicate,deadcode,perf`） |

维度标识符对照：

| 标识符 | 维度名称 | 分类 |
|--------|---------|------|
| `duplicate` | 重复代码 | 核心 |
| `deadcode` | 死代码/废弃资源 | 核心 |
| `perf` | 性能隐患 | 核心 |
| `structure` | 代码结构 | 核心 |
| `deps` | 依赖健康 | 核心 |
| `security` | 安全加固 | 扩展 |
| `observability` | 可观测性 | 扩展 |
| `config` | 配置管理 | 扩展 |
| `testing` | 测试覆盖 | 扩展 |
| `compat` | 兼容性 | 扩展 |

输出探测摘要：

```
项目: {项目名}
技术栈: {语言} + {框架}
扫描范围: {目录列表}
扫描维度: {N} 个（{维度名列表}）
```

## 第二步：多维度扫描

逐维度读取代码文件进行分析。每发现一个问题，记录：

- **维度**：所属维度
- **严重度**：Critical / High / Medium / Low
- **位置**：文件路径:行号
- **描述**：问题说明
- **建议**：优化方案

### 核心维度扫描要点

#### 1. 重复代码（duplicate）

- 相似函数体（逻辑相同但变量名不同）
- copy-paste 痕迹（多处几乎相同的代码块）
- 可提取为公共函数/组件的重复逻辑

#### 2. 死代码/废弃资源（deadcode）

- 未使用的 import/require
- 未调用的函数、未引用的变量
- 被注释掉的代码块
- 未使用的文件（组件、样式、图片、配置）
- package.json 中声明但未使用的依赖

#### 3. 性能隐患（perf）

- 循环内的重复计算或 I/O 操作
- 缺失的缓存/记忆化
- N+1 查询模式
- 不必要的同步阻塞操作
- 过大的包体积（未做 tree-shaking 或代码分割）

#### 4. 代码结构（structure）

- 过长函数（超过 50 行）
- 过深嵌套（超过 3 层）
- 职责不清的模块（一个文件做多件事）
- 缺失的抽象层（重复模式未封装）
- 过度耦合（模块间循环引用）

#### 5. 依赖健康（deps）

- 过时的依赖（主版本落后 2+ 个版本）
- 有已知安全漏洞的依赖
- 功能重叠的依赖（如同时有 lodash 和 underscore）
- 体积过大的依赖（有轻量替代品）
- 未使用的依赖

### 扩展维度扫描要点

#### 6. 安全加固（security）

- 硬编码的密钥、密码、token
- 未校验/转义的用户输入
- 路径遍历风险
- SQL/命令注入风险

#### 7. 可观测性（observability）

- 关键操作缺少日志
- 错误捕获后吞掉异常（空 catch 块）
- 缺少错误上下文信息

#### 8. 配置管理（config）

- 魔法数字/字符串
- 硬编码的 URL、端口、超时值
- 环境相关值未通过环境变量管理

#### 9. 测试覆盖（testing）

- 核心业务逻辑无单元测试
- 只有正常路径测试，缺少异常/边界用例
- 测试中硬编码的不稳定值（如时间、随机数）

#### 10. 兼容性（compat）

- 使用了已废弃的 API
- 平台特定代码无降级方案
- 未处理的版本差异

## 第三步：分析报告

将扫描结果以结构化格式输出（此步骤不写任何文件）：

```markdown
## 代码优化分析报告

项目: {项目名} | 技术栈: {语言+框架}
扫描范围: {目录} | 扫描维度: {N} 个
发现问题: {总数} 个（Critical: {n}, High: {n}, Medium: {n}, Low: {n}）

---

### Critical（{n}）

| # | 维度 | 位置 | 问题 | 建议 |
|---|------|------|------|------|
| 1 | 安全 | config.ts:12 | 硬编码的 API 密钥 | 迁移至环境变量 |

### High（{n}）

| # | 维度 | 位置 | 问题 | 建议 |
|---|------|------|------|------|
| 2 | 重复 | utils.ts:20, helpers.ts:45 | 两处相同的日期格式化逻辑 | 提取为 formatDate 公共函数 |

### Medium（{n}）
...

### Low（{n}）
...

---

### 维度统计

| 维度 | 问题数 | Critical | High | Medium | Low |
|------|--------|----------|------|--------|-----|
| 重复代码 | 3 | 0 | 2 | 1 | 0 |
| 死代码 | 5 | 0 | 1 | 2 | 2 |
| ... | ... | ... | ... | ... | ... |
```

## 第四步：用户确认

展示分析报告后暂停，等待用户确认。

明确告知用户可选操作：
- **全部采纳**：所有发现的问题均生成优化任务
- **按维度选择**：只处理特定维度的问题（如"只处理重复代码和死代码"）
- **按严重度选择**：只处理特定严重度以上的问题（如"只处理 High 及以上"）
- **逐项选择**：逐条确认是否处理
- **仅保留报告**：不生成任务，仅作参考

## 第五步：任务生成

将用户确认的优化项转化为 SuperPlanners 任务。

### 5.1 任务转化规则

每个优化项转化为一个任务，遵循原子化标准：

| 字段 | 转化规则 |
|------|---------|
| `id` | 按维度分组编号（如重复代码 "1", "1.1"；死代码 "2", "2.1"） |
| `title` | 动词开头，描述优化动作（如"提取日期格式化公共函数"） |
| `description` | 包含：问题描述 + 文件位置 + 具体优化方案 + 验收标准 |
| `priority` | Critical→critical, High→high, Medium→medium, Low→low |
| `estimate` | 根据问题复杂度估算（0.5h-4h） |
| `dependencies` | 同维度内有先后顺序的任务设置依赖 |

### 5.2 调用 MCP 工具创建计划

调用 `superplanners_plan` MCP Tool：

```
参数:
- requirement: "基于代码优化分析的改进计划 — {维度摘要}"
- project_name: "optimize-{原项目名或目录名}"
- tasks: [转化后的任务数组]
```

## 第六步：执行引导

任务计划创建成功后，输出执行建议：

```markdown
## 优化计划已创建

项目: {project_id}
任务数: {N} 个
预估总工时: {sum}h

### 建议执行顺序

1. **先处理 Critical 问题**（安全/阻断性问题优先）
2. **再处理高收益任务**（一次修复影响多处的问题）
3. **可并行任务**：{列出无依赖关系的任务组}

### 开始执行

查看任务：/superplanners:status {project_id}

执行时使用状态标记，系统会全自动更新进度：
- 开始任务：[TASK_STARTED: {project_id}/{task_id}]
- 完成任务：[TASK_COMPLETED: {project_id}/{task_id}]
```
